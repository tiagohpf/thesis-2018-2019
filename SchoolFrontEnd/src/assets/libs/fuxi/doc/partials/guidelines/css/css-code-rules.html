Regras de codificação

Esta secção apresenta as regras de codificação que a equipa segue para a declaração dos estilos.

Nomes de ID e Class

Não usamos nomes que reflitam questões apresentacionais nem atribuímos nomes sem qualquer significado. Procuramos sempre atribuír nomes que representem o propósito do elemento em questão ou, em alternativa, usar nomes genéricos.

Recomendado
Não recomendado

Prefixação dos nomes de ID e Class

Usamos um prefixo curto e percetível de acordo com o projeto, nomeadamente para distinguir os nossos estilos de outros de bibliotecas externas.

Recomendado
Não recomendado

Evitamos qualificar ID e Class

Procuramos ao máximo evitar a qualificação de ID e Class pois isso faz com que o estilo fique preso a um tipo de elemento. No exemplo não recomendado, caso a class .tabs fosse aplicado a uma ol o estilo não seria aplicado.

Recomendado
Não recomendado

Não especificamos unidades ao valor 0

Ao valor 0 nunca atribuímos nenhuma unidade de medida.

Recomendado
Não recomendado

Shorthand

Existem inúmeras propriedades CSS que podem ser alvo de shortand. Sempre que possível usamos esta técnica para manter a folha de estilos mais concisa.

Recomendado
Não recomendado

Omitimos o 0 antes dos valores decimais entre valores -1 e 1

Por não acrescentar valor à regra nunca colocamos o 0 antes dos valores decimais entre valores -1 e 1.

Recomendado
Não recomendado

Notação hexadecimal

Usamos notação hexadecimal para as cores e sempre que possível usamos apenas 3 caracteres. Preferimos esta notação pois facilita o copy/paste a partir dos editores de imagem no qual fazemos os protótipos.

Recomendado
Não recomendado

Delimitadores de nomes de ID e class

Separamos palavras do nome de uma classe com hyphen em vez de underscrore ou camel case. Preferimos esta opção pois a própria linguagem CSS utiliza esta notação para definir as suas propriedades. Por exemplo, font-size ou margin-top.

Recomendado
Não recomendado

Um seletor e uma propriedade por linha

Para facilitar a leitura do código declaramos um seletor e uma propriedade por linha.

Recomendado
Não recomendado

Ordenação alfabética das propriedades

Para facilitar a leitura do código ordenamos alfabeticamente as propriedades. Os vendor prefixs também são ordenados alfabeticamente. Colocamos a propriedade standard no fim dos vendor prefixs.

Recomendado
Não recomendado

Indentação do código

Indentamos as propriedades assim como os seletores como forma a que a hierarquia seja refletida, facilitando assim o entendimento.

Recomendado
Não recomendado

Terminamos todas as declarações com ;

Apesar de a última propriedade de uma declaração não necessitar de ser fechada com ; como forma de precaver erros futuros também a fechamos.

Recomendado
Não recomendado

Fim do nome da propriedade

Para facilitar a leitura do nosso código usamos um espaço entre os : e o valor da propriedade mas não usamos um espaço entre o nome da propriedade e os :

Recomendado
Não recomendado

Separação das declarações

Para facilitar a leitura do nosso código usamos um espaço entre o último seletor e a chaveta de abertura do declaration block. A chaveta de abertura fica sempre na linha do último seletor e a de fecho na linha abaixo da última propriedade.

Recomendado
Não recomendado

Comentários de secção e inserção de uma table of contents

Para facilitar a manutenção do código procuramos agrupar os estilos em zonas comentadas da folha de estilos e, no topo da mesma, colocamos a table of contents do ficheiro.
